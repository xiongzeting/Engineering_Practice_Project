# 工程实践中期检查报告

## 1. 项目基本信息
*   **项目名称**：数学公式识别与步骤评分系统
*   **项目目标**：利用计算机视觉（OCR）技术提取数学解题步骤，并结合大语言模型（LLM）与本地规则引擎，实现拟人化的“按步给分、错不株连”智能批改。
*   **报告阶段**：工程实践中期阶段

## 2. 目前已完成的工作与阶段成果

在工程实践的前半阶段，团队主要集中精力完成了系统的核心架构搭建、算法逻辑验证以及关键链路的打通。以下是详细进展：

### 2.1 核心后端服务开发 (Python/FastAPI)
1.  **OCR 识别与步骤提取模块** (`vision_ocr.py`, `step_parser.py`)：
    *   成功集成了图像预处理与版面分析能力，能够从用户上传的答卷图片中提取出包含数学公式的混合文本。
    *   实现了定制化的文本解析器 (`step_parser.py`)，能够根据换行、特定符号等数学推导特征，将一长段解题文本结构化切分为有序的独立解题步骤 (`StepItem`)。
2.  **双轨制智能评分引擎** (`scorer.py`)：
    *   **大模型 (LLM) 智能评分为主**：设计了专门针对数学批改的 Prompt 工程，引导 LLM 扮演“严格的数学老师”，按步判断逻辑正确性。实现了对 `/chat/completions` 和 `/responses` 接口的动态兼容与高可用调用。
    *   **本地规则评分兜底**：实现了一套基于基础合法性校验（如括号匹配、非法运算符检测）和参考答案字符串相似度（`SequenceMatcher`）的本地评分算法。当大模型服务不可用时，系统能无缝降级，保证评分流程不中断。
    *   **拟人化后处理策略**：为解决大模型评分偶尔过于死板的问题，开发了拦截器机制：
        *   **满分体验修正**：对于整体几乎全对的解答，系统会强制拉升至 100 分。
        *   **防连锁重罚**：首创性地实现了“错不株连”策略，一旦发现前面的关键错误（根错误），后续因沿用该错误导致的低分步骤将被兜底至及格线（如 60 分），并附带合理的评语。

### 2.2 前端交互界面开发 (HTML5/JS/CSS)
*   构建了直观的单页面应用（SPA）界面，实现了图片上传、OCR 结果与步骤预览的实时双向绑定。
*   集成了类似 LaTeX 的数学公式渲染方案，保证了复杂公式在网页上的正确展示。
*   开发了评分结果可视化面板，能够清晰地以表格形式展示每一步骤的内容、得分和 AI 评语，并提供整体的学习反馈。
*   添加了“评分诊断”高级折叠面板，方便在开发和排错阶段直观查看网络请求耗时、LLM 原始返回和错误日志。

### 2.3 工程化与质量保证文档
*   制定了详细的代码规范文档，统一了 Python 与前端代码的风格。
*   输出了《单元测试计划文档》和《集成测试计划文档》，为后期的质量保障和自动化测试落地奠定了理论基础。

## 3. 遇到的主要困难与解决方案

1.  **大模型返回 JSON 格式不稳定的问题**：
    *   *问题描述*：尽管 Prompt 中明确要求只返回 JSON，但某些大模型在实际调用中偶尔会附加 Markdown 代码块标记（如 ````json ... ````）或解释性前缀，导致标准 `json.loads` 解析失败。
    *   *解决方案*：在 `scorer.py` 中实现了 `_safe_json_parse` 函数，利用正则表达式 `\{[\s\S]*\}` 强制在任意文本中提取最大的花括号块进行解析，大幅提高了系统的容错率。
2.  **公式识别精度与排版问题**：
    *   *问题描述*：手写数学公式的复杂性（如上下标、分数线、积分号）导致 OCR 提取出的纯文本有时会丢失二维结构，增加步骤切分和比对的难度。
    *   *解决方案*：引入了人工微调干预环节（前端文本框可直接修改 OCR 结果），同时在评分引擎的 Prompt 中强调“宽容形式差异，注重逻辑本质”，并结合本地规则评分的相似度计算进行折中处理。未来考虑探索多模态直接推理。
3.  **大模型接口网络波动引起的超时报错**：
    *   *问题描述*：在使用第三方 API 打分时，偶发长达数十秒的阻塞，严重影响用户体验。
    *   *解决方案*：在代码层面强制加入了连接超时（Connect Timeout）和读取超时（Read Timeout）参数配置，结合本地规则引擎实现了优雅的回退（Fallback）机制。
