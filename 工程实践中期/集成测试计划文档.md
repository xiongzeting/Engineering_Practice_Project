# 集成测试计划文档 - 数学公式识别与步骤评分系统

## 1. 概述
本文档旨在为“数学公式识别与步骤评分系统”设计集成测试方案。集成测试的核心目标是验证系统各个功能模块（如图像预处理、OCR 识别、布局分段、步骤解析、打分引擎以及前端 API 交互）在**真实或类真实环境下的协同工作能力**。通过模拟用户从上传图片到获取评分结果的完整链路，确保模块间的数据传递与状态转换准确无误，系统在面临复杂或异常输入时依然具备高可用性与鲁棒性。

## 2. 测试策略与环境要求
*   **测试框架**: 推荐使用 `pytest` 配合 `FastAPI.testclient`（或 `requests`）进行后端 API 的集成测试；对于包含前端交互的全链路测试，可考虑引入 `Playwright` 或 `Selenium`。
*   **环境隔离**: 集成测试应在独立的测试环境（Test Environment）中运行，避免污染生产数据库或消耗真实的商业 API 额度（除非进行特定的灰度测试）。
*   **依赖服务打桩 (Stubbing/Mocking)**: 
    *   **非核心第三方服务**: 如认证系统（如果只是简单 Token 验证可 Mock）、外部存储（如对象存储可用本地文件系统替代）。
    *   **核心第三方服务 (LLM & OCR)**: 在日常持续集成 (CI) 中，建议对外部 LLM 服务（如 OpenAI/通义千问 API）和商业 OCR 服务进行 Mock，以保证测试的稳定性和速度。而在定期执行的回归测试或预发环境测试中，应接入真实的测试账号进行端到端联调。

## 3. 核心集成场景设计 (Integration Scenarios)

### 3.1 核心主链路：全功能正常运转 (Happy Path)
*   **场景描述**: 用户上传一张清晰的手写/打印数学解题图片，选择“智能评分”，并输入正确的参考答案，系统成功返回包含详细得分和反馈的评分报告。
*   **测试步骤**:
    1.  调用图片上传/识别 API（如 `/api/recognize`），传入标准的测试图片。
    2.  验证 API 成功返回提取出的文字和初步切分的步骤列表。
    3.  调用评分 API（如 `/api/grade`），传入步骤列表、图片 ID、参考答案，并勾选 `use_llm=True`。
    4.  验证 API 成功返回 200 状态码，且返回结果中 `ok` 为 `true`。
    5.  **断言核心数据**:
        *   `total_score` 在合理范围内（例如 80-100 分，取决于测试用例）。
        *   `steps` 数组长度与输入的步骤数量一致，且每个步骤均包含合理的 `score` 和 `reason`。
        *   `grading_meta.llm_used` 必须为 `true`。
        *   `feedback` 不为空。

### 3.2 降级/回退链路：LLM 服务不可用 (Fallback Path)
*   **场景描述**: 用户上传图片并请求智能评分，但此时大模型服务提供商宕机、网络超时或 API Key 失效。系统应能平滑降级，自动使用本地规则引擎给出合理的评分，保证用户体验不中断。
*   **测试步骤**:
    1.  准备合法的步骤数据。
    2.  在测试环境中，**主动修改配置或拦截网络请求**，使 LLM API 请求必定失败（例如抛出 Timeout 异常，或返回 401/500 状态码）。
    3.  调用评分 API，传入数据并勾选 `use_llm=True`。
    4.  验证 API 依然成功返回 200 状态码。
    5.  **断言核心数据**:
        *   系统正常返回总分和每个步骤的分数（基于规则计算）。
        *   `grading_meta.llm_used` 必须为 `false`（或标记为回退状态）。
        *   `grading_meta.scoring_mode` 应当是 `rule-based`。
        *   `grading_meta.llm_error` 应包含能够指示调用失败原因的错误信息（如“LLM评分未成功，已回退到规则评分”）。

### 3.3 异常输入处理链路 (Edge Cases & Exception Handling)
*   **场景 1：空数据或极少数据**
    *   上传完全空白的图片，验证 OCR 服务和步骤切分模块是否能妥善处理（如返回空步骤列表），随后评分 API 接收空步骤时，应返回合理的状态（如总分 0 或提示无法评分），而不应抛出 500 服务器内部错误。
*   **场景 2：超长文本攻击/极大图片**
    *   模拟前端传递极其冗长的 OCR 文本（超过几万字）或超清大图。验证系统是否具有合理的截断机制（如 `ocr_text[:2000]`）或文件大小限制，确保系统内存不会耗尽（OOM），且不引起外部 LLM 接口的 Token 超限报错（返回 400）。
*   **场景 3：恶意格式 (Malicious Input)**
    *   直接向评分 API 注入非预期类型的数据（如将 `steps` 字段从列表改为字符串，或包含恶意的脚本标签）。由于使用了 FastAPI（基于 Pydantic），应验证系统在路由层便能拦截此类请求并返回 422 Validation Error。

### 3.4 模块间协作特化测试 (Module Interaction)
*   **OCR -> 步骤解析器的边界测试**:
    *   输入包含大量复杂数学符号（如积分、矩阵、多行等式）的半结构化文本，验证 `step_parser` 能否将其合理切分成具有逻辑顺序的 `StepItem` 列表，而不是将半个公式截断。
*   **评分引擎策略组合测试**:
    *   **强干预策略验证**: 构造特定的步骤序列，使其在规则评分或被 Mock 的 LLM 评分下，触发“满分修正”（全对提分）或“防连锁重罚”（首错重罚，后续兜底 60 分）。验证经过 `_normalize_step_scores` 处理后输出给前端的最终分数和评语是否符合预期。

## 4. 执行计划与产出
*   **执行频率**:
    *   在每次合并到主干分支（如 `main` 或 `develop`）前，必须通过包含 Mock 的核心集成用例。
    *   在每次发版上线前（Release Candidate），需在预发环境（Staging）执行全量（包含连接真实 LLM 和 OCR 服务的端到端用例）回归测试。
*   **预期产出**:
    *   每次运行生成的自动化测试报告（如 Allure Report 或 pytest-html）。
    *   发现并在 Jira/Issue Tracker 中记录的模块集成缺陷。
    *   核心链路 API 响应时间统计，用于后续性能优化的基线参考。
