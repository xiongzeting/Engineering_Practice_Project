# 数学智能评分系统 - 编码与架构规范 (Coding Standards)

## 1. 概述
本规范旨在统一前后端团队的代码风格，提高代码的**可读性、可维护性和健壮性**。本项目后端基于 Python 3.10+，前端采用原生 JavaScript (ES6+) + HTML5 + CSS3。

---

## 2. Python 后端编码规范

### 2.1 代码格式与风格
*   **基础规范**：全面遵循 **PEP 8** 规范。建议项目中引入 `Black` 或 `Ruff` 作为自动格式化工具，单行代码长度限制建议为 120 字符。
*   **导入顺序**：
    1. 标准库导入 (如 `json`, `re`, `time`)。
    2. 第三方库导入 (如 `requests`)。
    3. 本地模块导入 (如 `app.config`, `app.schemas`)。
    *每个模块之间空一行，禁止使用 `from module import *`。*

### 2.2 命名规范
*   **变量与函数**：使用小写字母加下划线的蛇形命名法（`snake_case`），如 `step_scores`, `score_steps()`。
*   **私有函数/方法**：模块内不对外暴露的辅助函数，必须以单下划线开头，如 `_llm_score_steps()`、`_safe_json_parse()`。
*   **类名**：使用大驼峰命名法（`PascalCase`），如 `StepItem`, `StepScore`。
*   **常量**：使用全大写字母加下划线，如 `MAX_TIMEOUT_SEC`。

### 2.3 类型注解 (Type Hinting) - **强制要求**
*   所有函数**必须**包含参数和返回值的类型注解。
*   文件头部必须包含 `from __future__ import annotations` 以支持现代类型提示推导。
*   使用 Python 3.10+ 的联合类型语法（如 `str | None` 替代 `Optional[str]`）。
*   复杂数据结构需明确内部类型，如 `list[StepItem]`, `dict[str, Any]`。

**正确示例：**
```python
def _normalize_step_scores(raw_scores: list[dict[str, Any]], steps: list[StepItem]) -> list[StepScore]:
    # ...
```

### 2.4 异常处理与安全
*   **精准捕获**：尽量避免直接使用裸露的 `except Exception:`，除非在顶级调用链做兜底处理。
*   **安全解析**：处理外部不可控输入（如 LLM 返回的 JSON）时，必须使用 Try-Catch 或正则容错（参考现有的 `_safe_json_parse`）。
*   **字典取值**：从不确定的字典中获取数据时，强制使用 `.get("key", default)`，严禁直接使用 `dict["key"]` 以避免 `KeyError`。

## 3. 前端编码规范 (JS / HTML / CSS)

### 3.1 JavaScript (ES6+)
*   **变量声明**：优先使用 `const` 声明变量；仅在变量需要重新赋值时使用 `let`；**严禁使用 `var`**。
*   **DOM 操作**：
    *   DOM 元素的引用变量建议集中在文件顶部声明，并在变量命名上体现是 DOM 元素（如 `const ocrTextInput = document.getElementById("ocrText");`）。
    *   动态生成 DOM 树时，推荐使用 `document.createElement` 并逐级 `appendChild`，以避免内联拼接字符串带来的 XSS 注入风险（特别是渲染未知数学公式时）。
*   **异步请求**：统一使用 `async/await` 处理 Promise，取代旧的 `.then().catch()` 链式回调。
*   **判空与默认值**：多利用可选链操作符 (`?.`) 和空值合并操作符 (`??` / `||`)。如 `const text = (latex || "").trim();`。

### 3.2 HTML & CSS
*   **语义化标签**：合理使用 `<header>`, `<main>`, `<section>`, `<details>` 等 HTML5 语义化标签。
*   **CSS 命名**：
    *   Class 类名使用小写字母和中划线 (`kebab-case`)，如 `.math-panel`, `.math-fallback`。
    *   避免过度嵌套，选择器层级尽量不超过 3 级。
*   **CSS 变量**：颜色、间距等主题强相关的属性，统一抽取为 CSS 变量（如 `var(--line)`），以便后续支持暗色模式。

## 4. Git 提交规范

每次 Commit 应当是一个完整且单一的功能或修复，Commit Message 需遵循 [Angular 规范](https://www.conventionalcommits.org/zh-hans/v1.0.0/)：

**格式**：`<type>(<scope>): <subject>`

**常用 Type 列表**：
*   `feat`: 新增功能 (Feature)
*   `fix`: 修复 Bug
*   `refactor`: 代码重构（不增加新功能，也不修复 bug）
*   `style`: 代码格式修改（不影响逻辑，如删减空格、格式化）
*   `docs`: 文档更新
*   `perf`: 性能优化

**示例**：
*   `feat(scorer): 增加防连锁扣分的策略拦截器`
*   `fix(frontend): 修复公式无法正确换行渲染的问题`

## 5. 注释与文档

1.  **模块/文件级注释**：对于核心业务逻辑文件（如 `scorer.py`、`step_parser.py`），需在文件开头简要说明该文件的职责。
2.  **函数/方法注释**：复杂的业务函数（尤其是涉及算法和规则的函数）需提供 Docstring，说明参数含义、边界条件和返回值。
3.  **内联注释 (Inline Comments)**：
    *   代码逻辑显而易见时，不写无意义的注释（如 `i += 1 # i加1`）。
    *   当代码中使用了特殊的“业务策略”（如强制提分、正则回退）时，**必须**写明原因。例如：
        ```python
        # 策略1：若整体几乎全对，直接给满分，避免“全对却非100”的体验。
        if out and min(x.score for x in out) >= 95.0:
            # ...
