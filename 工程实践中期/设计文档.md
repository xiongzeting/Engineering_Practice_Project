# 智能数学步骤评分系统 - 核心设计文档

## 1. 项目概述
**项目名称**：数学公式识别与步骤评分系统
**项目目标**：通过计算机视觉（OCR）技术提取学生上传的数学解答图片中的公式与文字，将其按解题步骤进行结构化拆分，并利用大语言模型（LLM）与本地规则引擎相结合的方式，实现**“按步给分、错不株连”**的拟人化智能自动批改。

## 2. 系统整体架构
系统采用前后端分离架构，核心逻辑集中在后端服务层。

*   **前端交互层**：采用原生 HTML5/JS/CSS，提供友好的用户界面。包含图片上传、OCR 结果渲染（支持 LaTeX 公式展示）、步骤实时预览、人工修正以及评分结果的可视化呈现（步骤得分表格、整体反馈）。
*   **后端服务层 (Python)**：
    *   **视觉与识别模块** (`vision_ocr.py`, `layout_segmenter.py`)：负责图像预处理、版面分析与 OCR 文本提取。
    *   **后处理与纠错模块** (`ocr_corrector.py`, `step_parser.py`)：对 OCR 结果进行降噪、文本纠错，并按照数学推导逻辑将全文本段落拆分为独立的解题步骤 (`StepItem`)。
    *   **核心评分引擎** (`scorer.py`)：接收标准化步骤，执行双轨制打分策略，输出带有得分和具体评语的步骤报告 (`StepScore`)。

## 3. 核心业务流程 (Data Flow)
1.  **输入采集**：用户上传答题图片，并可选择性输入标准参考答案（Reference Solution）。
2.  **公式提取与结构化**：系统调用 OCR 服务识别图片，`step_parser.py` 将识别出的长文本依据换行、特定符号等特征，切分成有序的步骤数组集合。
3.  **人工干预（可选）**：前端展示切分结果，用户可在文本框中手动微调 OCR 的错漏字或不合理的切分。
4.  **智能评分请求**：前端将步骤列表与参考答案打包发送至 `/api/grade` 接口。
5.  **双轨评分执行**：评分引擎优先调用大语言模型进行逻辑判定；若未开启 LLM 或调用失败，则无缝降级至本地算法规则评分。
6.  **结果返回与渲染**：生成并返回总分、步骤得分表格及整体学习反馈。

## 4. 核心模块设计：评分引擎 (Scoring Engine)
评分引擎采用**“AI 评分为主 + 规则算法兜底”**的双轨制高可用设计。

### 4.1 LLM 智能评分策略
*   **Prompt 工程设计**：
    *   要求大模型扮演“严格的数学老师”。
    *   向模型输入完整的题目上下文（OCR 文本片段）以及结构化的步骤数组。
    *   注入明确的评分边界：“完全正确给100分，忌无理由给 90-99”、“首次严重错误重罚，后续步骤推导正确需手下留情”。
*   **高可用网络通信**：
    *   支持动态切换 API 路径（`/chat/completions` 与 `/responses` 互为备用机制），适配不同的大模型网关。
    *   支持流式（Stream）与非流式调用，并具备完整的超时截断（Timeout）与重试机制。
*   **结果解析与后处理 (Post-processing Interceptors)**：
    为弥补 LLM 偶尔出现的幻觉或严苛打分，系统实现了两套强干预拦截器：
    1.  **满分体验修正 (Strategy 1)**：若系统判定所有步骤的最低分仍 $\ge$ 95 分，拦截器会强制将所有步骤提至 100 分，并修正评语为“步骤正确，给满分”，优化用户的满分正反馈。
    2.  **防连锁重罚 (Strategy 2)**：遍历步骤序列，若发现首个重大错误（分数 $< 60$ 的“根错误”），拦截器会自动将其后方所有低于 60 分的步骤兜底提升至 60 分，并修改评语为*“受前序错误影响，当前步不重复重罚”*，完全拟合人类教师的阅卷习惯。

### 4.2 本地规则评分机制 (Fallback Rule-based Scoring)
作为系统的保底策略，当 LLM 服务不可用时触发，其得分由两部分动态加权计算（0.55 $\times$ 基础合法性 + 0.45 $\times$ 相似度）：
*   **基础合法性检验 (Basic Validity, 占比 55%)**：
    校验数学表达式的完整性。如：左右括号不匹配扣 20 分；缺少数字/等号扣 15 分；存在连续非法运算符（如 `*=`、`++`）扣 10 分。
*   **参考答案相似度检测 (Similarity, 占比 45%)**：
    若用户提供了参考解法，使用 `SequenceMatcher` 计算当前步骤与各个参考步骤的文本相似度（Ratio），取最大匹配度转化为百分制分数。如果相似度 $< 0.35$，判定为“差异较大”，$< 0.65$ 判定为“部分一致”。无参考解法时，该项默认赋及格分 75 分。

## 5. 异常处理与诊断日志
*   **评分诊断面板 (Diagnostic Meta)**：每次请求均会收集耗时 (`elapsed_ms`)、HTTP 状态码、失败原因及 Response 截断预览，一并返回给前端的“评分诊断”折叠面板中，极大地降低了模型 API 的排错成本。
*   **容错解析**：针对 LLM 偶尔不按规范输出 JSON 的问题，后端利用正则表达式 `\{[\s\S]*\}` 强制提取文本块中的 JSON 字符串并进行安全反序列化 (`_safe_json_parse`)。

## 6. 未来优化与扩展方向
1.  **公式语义理解**：当前规则评分采用纯文本相似度（SequenceMatcher），未来可引入 SymPy 符号计算库，判断多项式展开、等式移位在数学上是否严格等价（如 $x+y=1$ 等价于 $y=1-x$）。
2.  **多模态大模型直出**：未来可跳过传统 OCR 切分阶段，直接将答卷图片喂给 GPT-4o 等多模态模型，通过设定的 Prompt 一次性输出步骤切分与评分结果，进一步缩短业务链路。
