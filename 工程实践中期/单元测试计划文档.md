# 单元测试计划文档 - 数学公式识别与步骤评分系统

## 1. 概述
本文档旨在为“数学公式识别与步骤评分系统”的后端核心逻辑编写单元测试计划。通过单元测试，验证各模块（尤其是核心评分引擎）在不同输入情况下的正确性、鲁棒性以及异常处理能力，确保后续功能迭代不会破坏现有逻辑。

## 2. 测试框架与工具
*   **测试框架**: `pytest` (Python 环境下最成熟的单元测试框架)
*   **Mock 工具**: Python 内置的 `unittest.mock` (用于拦截和模拟外部大模型 API 请求，防止测试产生额外费用或因网络波动失败)
*   **覆盖率统计**: `pytest-cov` (目标覆盖率：核心服务层 > 85%)

## 3. 测试范围重点
本次单元测试主要聚焦于后端服务层的核心计算和逻辑解析模块，重点文件为：
*   `app/services/scorer.py` (核心评分逻辑，包含规则引擎与 LLM 数据后处理)
*   （视进度可扩展）`app/services/step_parser.py` (步骤拆分逻辑)

## 4. 核心测试用例设计 (`scorer.py`)

### 4.1 基础合法性评分 (`_basic_validity_score`)
*   **用例 1.1 (正常表达式)**：输入 `x^2 + 2x + 1 = 0`，预期得分 `100.0`。
*   **用例 1.2 (括号不匹配)**：输入 `(x + 1 = 2` (少右括号)，预期扣 20 分，得分 `80.0`。
*   **用例 1.3 (缺失数字与等号)**：输入单纯的字母组合如 `abc`，预期扣 15 分，得分 `85.0`。
*   **用例 1.4 (连续非法运算符)**：输入 `x ++ 1 == 2`，预期扣 10 分。
*   **用例 1.5 (组合错误)**：综合以上多个错误，验证得分下限（需确保 `max(score, 0.0)` 生效，不出现负分）。

### 4.2 规则兜底评分逻辑 (`_rule_score_steps`)
*   **用例 2.1 (无参考答案)**：测试仅依赖基础合法性评分的情况，验证相似度部分默认得分(75分)及权重计算。
*   **用例 2.2 (高度相似)**：提供与被测步骤几乎一样的参考答案，预期相似度得分接近 100，总评语为“与参考解法高度一致”。
*   **用例 2.3 (差异巨大)**：提供毫不相关的参考答案，预期相似度 $< 0.35$，评语出现“差异较大，建议检查推导或计算”。

### 4.3 LLM 评分结果后处理与策略 (`_normalize_step_scores`)
*   **用例 3.1 (满分修正策略)**：
    *   输入：LLM 返回的步骤分数数组为 `[95, 96, 98]`。
    *   预期输出：所有分数被强制修正为 `100.0`，评语更新为“步骤正确，给满分”。
*   **用例 3.2 (防连锁重罚策略)**：
    *   输入：LLM 返回的步骤分数数组为 `[100, 50, 40, 30]` (第2步发生了根错误，导致后续极低分)。
    *   预期输出：后续连续低分被兜底，输出变为 `[100.0, 50.0, 60.0, 60.0]`，且后两步评语被修改为“受前序错误影响，当前步不重复重罚”。
*   **用例 3.3 (乱序或缺漏步骤处理)**：
    *   输入：LLM 漏掉了第 2 步的评分。
    *   预期输出：系统为第 2 步默认赋中性分 `70.0`，并提示“模型未返回该步评分”。

### 4.4 JSON 安全解析容错 (`_safe_json_parse`)
*   **用例 4.1 (标准 JSON)**：输入合法的 JSON 字符串，预期正确解析为 Dict。
*   **用例 4.2 (Markdown 包装的 JSON)**：输入被 ````json { ... } ```` 包裹的字符串，预期能使用正则准确提取大括号内的内容并解析。
*   **用例 4.3 (完全非法的文本)**：输入纯口语文本或截断的 JSON，预期返回 `None` 且不抛出异常导致进程崩溃。

### 4.5 主流程入口 (`score_steps`)
*   **用例 5.1 (启用 LLM 且调用成功)**：验证返回的 `scoring_mode` 为 `llm+rule`，且包含 `meta` 诊断信息。
*   **用例 5.2 (启用 LLM 但 API 失败，触发回退)**：Mock `requests.post` 抛出 Timeout 异常，验证系统能平滑回退，返回规则评分的结果，且 `scoring_mode` 为 `rule`，并在 `llm_error` 中包含超时错误说明。

## 5. Mock 策略执行方案
针对大模型评分请求，严禁在 CI/CD 流水线中直接发起真实的网络请求。需在 `tests/test_scorer.py` 中使用如下策略：

```python
from unittest.mock import patch

@patch('app.services.scorer.requests.post')
def test_llm_score_steps_success(mock_post):
    # 设定 Mock 对象的返回 JSON 和状态码
    mock_post.return_value.status_code = 200
    mock_post.return_value.ok = True
    mock_post.return_value.json.return_value = {
        "choices": [{"message": {"content": "{\"step_scores\":[{\"index\":1,\"score\":100,\"reason\":\"正确\"}],\"total_score\":100,\"feedback\":\"很好\"}"}}]
    }
    
    # 执行测试逻辑
    # ...
```

## 6. 测试执行与集成
*   **本地执行命令**: `pytest tests/ -v --cov=app/services`
